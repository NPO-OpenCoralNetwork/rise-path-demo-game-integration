## 🛠️ コントリビュートの方法

### 1. バグの報告
バグを発見した場合は、[GitHub Issues](issues) から報告してください。その際、以下の情報を含めていただけると修正がスムーズになります：
- 再現手順
- 期待する挙動と実際の挙動
- 実行環境（ブラウザ、OS、Node.jsバージョン）
- 関連するエラーログやスクリーンショット

### 2. 機能の提案
新しい機能や改善のアイデアがある場合は、まず Issue を作成してディスカッションを開始してください。

### 3. コードでの貢献（バグ修正・機能追加）
1. 既存の Issue から、ご自身が担当したいタスクを選択してください。
   - 初心者の方は、**`good first issue`** ラベルのついたタスクからのスタートを推奨します。
2. 担当するタスクの Issue に「着手します」とコメントを残してください（重複作業を防ぐため）。
3. 後述する **「Git ブランチ & コミットルール」** に従って作業を進めてください。

## 🚀 ローカル開発環境のセットアップ

1. **リポジトリのクローン**:
   ```bash
   git clone https://github.com/NPO-OpenCoralNetwork/rise-path-demo-game-integration.git
   cd rise-path-demo-game-integration
   ```
2. **環境変数の作成**:
   ルートディレクトリに `.env.local` ファイルを作成し、以下を設定します。
   ```env
   # ローカルLLM（Hermes）の設定（標準）
   HERMES_API_URL=http://127.0.0.1:8642
   HERMES_API_KEY=change-me-local-dev
   ```
3. **起動**:
   ```bash
   npm install
   npm run dev
   ```

## 🌿 Git ブランチ & コミットルール

安全でクリーンな履歴を保つため、コミットとブランチ作成の規則を統一しています。

### ① ブランチ作成ルール
必ず `main` から作業用のトピックブランチを作成してください。
- **命名規則**:
  - 新機能追加: `feature/機能名` (例: `feature/add-stage-3`)
  - バグ修正: `fix/修正内容` (例: `fix/compilation-error`)
  - ドキュメント修正: `docs/修正内容` (例: `docs/update-readme`)
  - アセット追加: `asset/アセット名` (例: `asset/slime-image`)

### ② コミットメッセージの規則
以下のプレフィックスを使用してください。
- `feat:` 新しい機能やロジックの追加
- `fix:` バグや表示崩れの修正
- `docs:` ドキュメントの修正のみ
- `asset:` ゲーム画像、BGM、音声の追加・変更
- `refactor:` 動作は変えないコード整理

*例:* `asset: スライム画像を透過済みの新しいアセットに差し替え`

## 📤 プルリクエスト（PR）の提出ルール

1. **提出前のチェック**:
   - [ ] `npm run build` を実行して、エラーなくビルドが通ること。
   - [ ] 機密情報（本物のAPIキーなど）が `.env.local` 以外のファイルにハードコードされていないこと。
   - [ ] アセットを追加する場合、ガイドライン（サイズ圧縮、透過PNG）に準拠していること。
2. **PRの作成**:
   - PRのタイトルには対応する Issue 番号（例: `Fixes #17`）を含め、関連するタスクを自動でクローズできるようにしてください。
3. **レビューとマージ**:
   - 提出された PR は、チームメンターまたは他の開発者によってレビューされます。
   - 最低1名以上の **Approve（承認）** を受けてから `main` ブランチへマージされます。
