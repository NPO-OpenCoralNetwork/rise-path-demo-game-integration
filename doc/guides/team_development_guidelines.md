# 👥 チーム開発ガイドライン

本ドキュメントは、プロジェクトメンバーがスムーズに共同開発を進め、コードや設計仕様、ビジュアルデザインのすれ違いを防ぐための基本ルールと運用プロセスを定義したものです。

---

## 1. デザインツール（Figma/Miro）とドキュメントの連携

チーム内で「どの画面が最新か分からない」「アセットの場所が不明」といった問題を解決するためのルールです。

### ① 「ディープリンク（個別リンク）」の徹底
*   仕様書やタスク票にFigmaやMiroのリンクを貼る際は、全体URLではなく、必ず**「該当するフレームやオブジェクト」を直接指すディープリンク**を貼ってください。
*   **Figmaでの取得方法**: 
    1. 該当のフレーム（例: `バトルUI`）をクリックして選択。
    2. 画面右上の「共有 (Share)」をクリック、または `Cmd + L` (Mac) / `Ctrl + L` (Win) で選択オブジェクトのリンクをコピー。
*   **Miroでの取得方法**:
    1. オブジェクトやフレームを右クリック。
    2. `Copy link` を選択。

### ② ドキュメント内のプレビュー埋め込み
*   NotionなどのツールにFigma/MiroのURLを記述する際は、単なるテキストリンクではなく、「埋め込み（Embed）」機能を利用してドキュメント内で直接ビジュアルを確認できるようにしてください。

## 2. Git ブランチ運用とコミットルール

共同でプログラムやアセットを管理する上での基本的な運用ルールです。

### ① ブランチ作成ルール
*   `main` ブランチに直接コミット（プッシュ）することは避け、必ず作業用ブランチを作成してプルリクエスト（PR）を経由してマージします。
*   **ブランチの命名規則**:
    *   新機能・画面の追加: `feature/機能名` (例: `feature/add-stage21`)
    *   バグや表示の修正: `fix/修正内容` (例: `fix/battle-bgm-bug`)
    *   ドキュメントの修正: `docs/修正内容` (例: `docs/update-gdd`)
    *   アセットのみの追加: `asset/アセット名` (例: `asset/new-boss-image`)

### ② コミットメッセージのプレフィックス
コミットログを分かりやすく整理するため、以下のコミットメッセージのプレフィックスを使用してください。

*   `feat:` 新しい機能や画面、ロジックの実装
*   `fix:` バグや動作不良、レイアウト崩れの修正
*   `docs:` 設計書、仕様書、READMEなどドキュメントのみの追加・修正
*   `asset:` ゲーム用画像、音声、BGMなどのリソース追加・変更
*   `refactor:` 動作は変えずにコードの構造を整理
*   `chore:` ライブラリのインストールや設定ファイルの修正など

*コミットメッセージの例:*
> `asset: ステージ3用の背景画像(snow.jpg)を新しいものに差し替え`
> `fix: battle.js内の敵キャラクターのスケール値を調整`

### ③ 競合 (Conflict) の回避
*   作業を開始する前に、必ず `git checkout main` -> `git pull` を行い、最新の `main` をローカルに取り込んでから作業用ブランチを切ってください。
*   特にアセットファイルや共通設定ファイルの変更は競合しやすいため、小まめなコミットと、マージ前の `git merge main` の実行を心がけてください。

## 3. アセットファイルの容量管理と Git LFS

ゲームの表示速度（ロード時間）を高速にし、リポジトリの容量肥大化を防ぐためのルールです。

### ① 画像・音声の圧縮ルール
- **画像の圧縮**:
  - 背景画像やキャラクターの画像は、見た目の品質を落とさない範囲で適切に圧縮してください（WebPの利用や、TinyPNGなどの圧縮ツールの活用を推奨）。
  - 必要以上に巨大な画像（例: 横幅 4000px を超える背景など）は、ロード遅延の原因になるため、ゲーム内の表示サイズに合わせてあらかじめリサイズ（例: フルHDサイズ 1920x1080 等）してからコミットしてください。
- **音声の圧縮**:
  - 効果音（SE）やBGMなどのmp3/wavファイルは、ビットレートを適切に調整（例: BGMは 128kbps〜192kbps、SEはモノラル化など）し、ファイル容量を最小限に抑えてください。

### ② Git LFS (Large File Storage) の適用
- 容量の大きいバイナリファイル（**2MBを超える画像や音声**）をリポジトリに追加する場合は、Gitリポジトリ全体の肥大化を防ぐため、必ず `Git LFS` を通して管理してください。
- すでに `.gitattributes` で設定されているファイル形式（`.png`, `.jpg`, `.mp3` など）は自動でLFS管理下に置かれますが、新しく大容量ファイルをコミットする際は `git lfs track` の設定状況を確認してください。

## 4. プルリクエスト（PR）のレビュー・マージプロセス

安全でクリーンな共同開発を行うため、以下のPR承認フローを遵守してください。

### ① PR作成のチェックリスト
PRを提出する前に、作成者は以下を確認してください：
- [ ] ローカルでゲームを起動し、エラーなく動作すること。
- [ ] 追加・変更したアセットが、ガイドライン通りのフォーマット（透過、サイズ等）であること。
- [ ] 意図しないデバッグログ（`console.log` 等）や一時ファイルが含まれていないこと。

### ② 承認・マージの基準
- **コード変更（ロジック追加等）**:
  - 最低1名以上の他の開発者から **Approve (承認)** を得た上で、マージ（またはSquash merge）を行ってください。
- **アセット変更（画像・音声）**:
  - 新規アセットや差し替えがある場合は、デザイナーおよび実装担当者がローカル（実機）での見え方・聞こえ方を確認し、承認した上でマージしてください。
