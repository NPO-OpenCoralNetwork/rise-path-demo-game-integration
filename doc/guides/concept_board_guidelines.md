# 🎨 コンセプトボード作成ガイドライン

P-School チームがMiro上でコンセプトボードを作成・運用する際の共通ルールです。

---

## 1. 基本原則

### 🎯 3つの原則

| 原則 | 説明 |
|---|---|
| **明確さ** | 「このボードは何を伝えたいのか」が**3秒**で分かること |
| **視覚優先** | テキストではなく画像・図・色で語ること |
| **生き物として運用** | 作って終わりではなく、MTGの度に参照・更新すること |

### ⚠️ 絶対ルール
- **1ボード1テーマ**: あれもこれも詰め込まない。テーマが増えたらボードを分割する
- **テキストは補足のみ**: 画像をメインにし、テキストは短い注釈に留める
- **余白（ネガティブスペース）を残す**: 全面を埋め尽くさない。視覚的な「呼吸」を確保する

---

## 2. ビジュアルヒエラルキー（視線誘導）

ボードを見た人の視線を**意図した順番**で誘導するための設計ルールです。

| テクニック | やり方 | 例 |
|---|---|---|
| **ヒーロー画像** | 最も重要なビジュアルを1つ、大きく中央に配置する | ゲームのメインビジュアル |
| **サイズの差** | 重要なものは大きく、補足は小さく | メインキャラは大、サブアセットは小 |
| **色のコントラスト** | 注目させたい要素に鮮やかな色を使う | 🚨ルール=赤、✅完了=緑 |
| **視線の流れ** | F字型 or Z字型のレイアウトで自然に読ませる | 左上→右上→左下→右下 |

---

## 3. 付箋の色ルール（カラーコード）

チーム全体で付箋の色の意味を統一します。

| 色 | 用途 | Miro色名 |
|---|---|---|
| 🔴 **赤 / ピンク** | 絶対ルール・警告・ブロッカー | `light_pink` |
| 🟢 **緑** | Git/開発ルール・完了済み | `light_green` |
| 🔵 **青** | 参考資料・リンク・ドキュメント | `light_blue` |
| 🟡 **黄色** | 通常メモ・アイデア・議事録 | `light_yellow` |
| 🟠 **オレンジ** | 進行中・作業中タスク | `orange` |
| ⚫ **ダークブルー** | セクションタイトル・見出し | `dark_blue` |
| ⬜ **グレー** | 未着手・保留・アーカイブ | `gray` |

## 4. ボード作成の5ステップ

新しいコンセプトボードを作る際は、以下の手順に従ってください。

### Step 1: 目的の明確化
- 「何のために作るのか」「誰にどういう印象を与えたいのか」を言語化する
- ボード最上部にタイトルと1行の目的文を記載する

### Step 2: キーワード抽出
- ターゲットや提供したい価値をキーワードとして書き出す
- 例: `ドット絵` `RPG` `教育ゲーム` `ブロックプログラミング` `ファンタジー`

### Step 3: 素材収集
- キーワードに基づき、画像・配色（カラーパレット）・フォント・テクスチャを集める
- Pinterestでのアイデア収集 → Miroへの整理が推奨フロー

### Step 4: 整理と構築
- 集めた素材をボードにコラージュ
- グループ分けをして名前をつける（Miroの「Frame」機能を使う）

### Step 5: 調整と洗練（ステップバック・テスト）
- 全体を見て、コンセプトから外れているものがないかチェック
- **数分離れてから戻り、最初に目が行く場所がコア・コンセプトかどうか確認する**
- そうでなければ、サイズ・色・配置を調整する

## 5. ボードの構成テンプレート

P-School のMiroボードは、以下のエリア構成を標準とします。

```
+-----------------------------------------------------------------------------------+
|  ボード最上部: タイトル＋ヒーロー画像（ゲームメインビジュアル）                     |
+-----------------------------------------------------------------------------------+
|                                                                                   |
|  +---------------------------+       +-----------------------------------------+  |
|  | ① チーム開発ルール         |       | ② アセット進行管理カンバン              |  |
|  |  (赤/緑の付箋でルール明示) |       |  (未着手 → 制作中 → 適用完了)          |  |
|  +---------------------------+       +-----------------------------------------+  |
|                                                                                   |
|  +---------------------------+       +-----------------------------------------+  |
|  | ③ ムードボード（世界観）    |       | ④ キャラクター＆モンスター図鑑          |  |
|  |  (参考画像/カラーパレット) |       |  (Before/After + プロンプト)            |  |
|  +---------------------------+       +-----------------------------------------+  |
|                                                                                   |
|  +---------------------------+       +-----------------------------------------+  |
|  | ⑤ AI生成プロンプト集       |       | ⑥ ゲーム画面遷移図（Mermaid図）        |  |
|  |  (スタイル別テンプレート)  |       |  (フローチャート + アセットマップ)       |  |
|  +---------------------------+       +-----------------------------------------+  |
|                                                                                   |
|  +---------------------------+                                                    |
|  | ⑦ 意思決定ログ             |                                                    |
|  |  (日付入りで方針変更を記録)|                                                    |
|  +---------------------------+                                                    |
+-----------------------------------------------------------------------------------+
```

### 各エリアの説明

| エリア | 目的 | 主な要素 |
|---|---|---|
| **① チーム開発ルール** | 全員が守るべきルールを視覚的に共有 | アセット上書きルール、Git命名規則 |
| **② アセット進行管理** | アセット差し替えの進捗をカンバンで管理 | 未着手/制作中/完了の3列 |
| **③ ムードボード** | ゲームの世界観・雰囲気を定義 | 参考画像、カラーパレット、フォント見本 |
| **④ キャラクター図鑑** | 各キャラの旧→新素材を比較表示 | Before/After画像、AI生成プロンプト |
| **⑤ AI生成プロンプト集** | NanoBanana Pro 2のテンプレート集 | ドット絵/手描き/フラットアイコンの型 |
| **⑥ ゲーム画面遷移図** | 画面遷移とアセット使用箇所のマッピング | Mermaid図、フローチャート |
| **⑦ 意思決定ログ** | デザイン方針の変更理由を記録 | 日付+決定内容+理由の付箋 |

## 6. やってはいけないこと（NG集）

| ❌ NG | ✅ 代わりにやるべきこと |
|---|---|
| テキストだらけにする | 画像メインにし、テキストは短い補足のみ |
| 素材を詰め込みすぎる | 余白を意識的に残す |
| 画像だけで意図を説明しない | 「なぜこの画像を選んだか」を短いコメントで添える |
| 1つのボードに全部載せる | テーマごとにボードを分割する |
| 作って終わりにする | MTGで「これはコンセプトに合うか？」の判断基準として活用 |
| 付箋の色をバラバラに使う | 上記のカラーコード表に従って統一する |

## 7. 運用ルール

### 更新タイミング
- **スプリント開始時**: カンバンのタスク更新、新しいアセット要件の追加
- **デザインレビュー時**: Before/After画像の更新、意思決定ログの追記
- **新メンバー参加時**: ムードボードとルールエリアを使ったオンボーディング

### アクセス権限
- チームメンバー全員に**編集権限**を付与する
- 外部共有時は**閲覧のみ**のリンクを発行する

### 意思決定ログの書き方
```
📅 2026-07-03
📝 決定: キャラクターのアート方向をドット絵16bitスタイルに統一
💬 理由: 教育ゲームとしての親しみやすさと、開発コストの低減を両立するため
👤 決定者: チーム全員（スプリント会議にて）
```

## 8. Markdown (.md) でのデザインコンセプト記述ルール

Miroボードのような視覚的ツールに加え、リポジトリ内のMarkdown（`.md`）ファイルでデザインコンセプトや世界観、仕様をまとめる際の記述ルールです。

### 💡 伝わるMarkdownにするための5つの工夫

#### 1. ビジュアルのインライン配置
言葉だけでは伝わりにくい色彩設計やアートスタイルは、必ず画像をインラインで埋め込んでください。
- **書き方**: `![キャプション（何を表す画像か）](画像ファイルの相対パス)`
- **例**: `![ドット絵スライムのカラーバリエーション](../assets/slime_colors.png)`

#### 2. Mermaidによる図解
遷移フローやデータの関連図は、画像で書き出すのではなくMarkdownコードとして記述し、変更履歴（Git Diff）を追いやすくします。
- **書き方**: ` ```mermaid ` で開始し、フロー図を作成します。
- **活用例**: 画面遷移、アセットの読み込みタイミング、キャラクターの状態遷移など。

#### 3. GitHubスタイルの「警告（Alerts）」の活用
開発メンバーやデザイナーに見落としてほしくない重要な決定事項は、引用符を使った警告表示で目立たせます。
- **特に重要なデザイン制約**: `> [!IMPORTANT]`
- **間違えやすい仕様・禁止事項**: `> [!WARNING]` または `> [!CAUTION]`

#### 4. コピペ可能なコードブロックの提供
画像生成AIのプロンプト、CSSで指定するカラーコード（HEX/RGB）、Phaser等での変数名は、コピー＆ペーストしやすいように独立したコードブロック（` ``` `）で囲みます。
````markdown
🎨 **プライマリカラー（メインテーマ）**
`#4CAF50` (Vibrant Green)

🤖 **NanoBanana用プロンプト案**
```text
pixel art sprite of a slime, 16-bit retro RPG style, vibrant colors, solid flat white background, clean edges, centered, full body, isolated, detailed shading --aspect 1:1
```
````

#### 5. 相互リンクによる情報の網羅
他の設計仕様書や、関連するMiroボードのURLなどを適切にリンク配置します。
- **リポジトリ内ドキュメントリンク**: [ファイル名](相対パス/ファイル名.md)
- **Figma/Miroへのリンク**: [Miroボード](URL)
